# 皇冠虚拟余额账号自动投注设计

**日期：** 2026-07-13
**状态：** 待用户书面确认
**适用范围：** 当前皇冠抓水投注程序中的监控、投注规则、投注账号、自动投注 worker

## 1. 目标

在现有程序内补全自动投注链路：Watcher 持续读取皇冠当前开放盘口；赔率或盘口变化命中已启用的投注规则后，程序重新确认当前盘口和赔率，按规则目标总金额选择已启用账号并拆分金额，通过皇冠网页使用的 `FT_order_view` 和 `FT_bet` API 创建注单。

本阶段完成标准是能够从真实监控变化触发规则，并在用户提供的不可充提、虚拟余额账号上生成皇冠注单。只确认投注触发和注单创建，不等待比赛结算，也不在本程序内计算回测收益率。

## 2. 已确认的业务决定

| 项目 | 决定 |
|---|---|
| 账号性质 | 当前及后续添加的账号均按不可充提、仅含虚拟余额的皇冠测试账号处理 |
| 程序结构 | 改造现有投注功能，不新建另一套 Demo runtime，不另起一套账号或规则页面 |
| 触发入口 | Watcher 发现盘口或赔率变化，且命中已启用投注规则 |
| 市场范围 | Watcher 已规范化支持的赛前/滚球、让球/大小球、全场/半场及盘口变体 |
| 投注方式 | 直接调用皇冠 API；预览使用 `FT_order_view`，提交使用 `FT_bet` |
| 目标金额 | 使用规则配置的目标投注总金额 |
| 余额 | 执行前刷新账号网站余额；皇冠返回的最新 `reportedBalance` 即该账号的可执行虚拟余额 |
| 单笔上限 | 使用每个账号自行配置的 `perBetLimit`；当前用户配置为每账号 50 CNY |
| 拆分方式 | 按账号投注顺序拆分规则目标总金额；每个账号在同一批次最多提交一个子注单 |
| 拒绝处理 | 当前账号明确拒绝后立即解除锁，剩余金额换下一个可用账号；没有账号则放弃未完成部分 |
| 成功处理 | 收到明确成功结果和注单标识后记账并解除账号锁；该账号可用于后续规则批次 |
| 状态未知 | 超时、断线或响应无法确定时不自动重投，保留账号锁，防止重复下注 |
| 结算 | 不等待、不抓取比赛结算；网站自带回测和收益率统计继续由网站负责 |
| UI | 不新增真实资格、金额预算、短期授权的完整操作面；继续使用规则、账号和运行控制台 |

## 3. 非目标

- 不支持真实入金、提现或真钱余额管理。
- 不替用户判断策略好坏，不自动调整规则或目标金额。
- 不等待赛果，不在本程序内计算盈亏、ROI 或回测曲线。
- 不以浏览器点击作为正常投注路径，也不在 API 失败时自动切换成浏览器点击。
- 不绕过皇冠登录、盘口关闭、赔率变化、金额上下限或其他服务端拒绝。
- 不为未知结果自动重试 `FT_bet`。
- 不把 Windows 发布改造混入投注业务代码；远程更新删除和桌面快捷方式由独立规格 `2026-07-13-crown-manual-portable-and-desktop-shortcut-design.md` 处理。

## 4. 现状与需要替换的阻断

现有程序已经具备规则命中、批次账本、账号排序、账号锁、fencing、幂等键、accepted/rejected/unknown 状态和 worker 框架，但生产 Provider 尚未提交真实请求：

- `CrownAccountExecutionProvider.submit()` 当前固定返回 `crown-submit-transport-unavailable`。
- `FT_order_view` 的生产路径仍被 canonical capability matrix 阻断。
- runtime 目前强制要求 preview/submit/reconciliation 三项全部可用；本阶段不等待结算，因此不能继续把 reconciliation 当作启动前提。
- `reportedBalance` 当前只作为账号检测信息展示，没有进入 stake allocation。
- Watcher 独立子进程退出时缺少可观察退出原因和受控自动恢复。
- 登录成功时间只在初次登录时写入；后续 session 校验成功未刷新时间，导致运行控制台可能误报 `monitor-login-not-fresh`。

改造保留已有账本、锁和幂等语义，只替换上述未接通或不符合当前测试账号业务的部分。

## 5. 总体链路

```text
皇冠盘口变化
  -> Watcher 规范化并写入当前盘口状态
  -> 已启用规则匹配
  -> 创建唯一投注信号/批次
  -> 重新读取同一赛事、市场、方向、盘口和最新赔率
  -> 按账号顺序分配目标总金额
  -> 锁定一个账号并刷新登录、虚拟余额
  -> FT_order_view 校验盘口、赔率和服务端金额边界
  -> FT_bet 只提交一次
  -> accepted: 保存注单并解锁
     rejected: 保存拒绝并解锁，剩余金额换账号
     unknown: 保存未知并保持锁，不重试
  -> 没有更多账号时，将剩余金额记为 unfilled
```

同一规则变化只能创建一个稳定批次；同一批次、同一账号和同一子注单不能因页面刷新、worker 重启或重复消费再次调用 `FT_bet`。

## 6. 规则匹配与当前盘口复核

### 6.1 触发规则

继续使用现有动态投注规则卡和 Watcher 信号。规则必须满足：

- 规则已启用；
- 联赛属于该规则选择范围；
- 赛前/滚球模式一致；
- 市场类型、比赛阶段、方向、盘口和目标赔率范围一致；
- 当前赛事仍在 `monitor_event_state.active=1` 的皇冠开放盘口中。

“今日比赛”继续表示皇冠当前开放盘口，不按自然日或未来 24 小时过滤。

### 6.2 提交前复核

规则信号只用于创建批次，不能直接作为提交参数。每个子注单提交前必须从当前监控投影重新锁定：

- 皇冠赛事标识；
- 联赛和球队；
- 赛前/滚球；
- 全场/半场；
- 让球/大小球；
- 下注方向；
- 盘口值；
- 最新赔率；
- 数据版本和采集时间。

任何标识缺失、市场已关闭、模式变化、方向漂移、盘口变化或赔率超出规则边界，都在调用 `FT_bet` 前终止该批次未提交部分。不能根据字段名、ID 前缀或相邻盘口猜测映射。

## 7. 账号选择、余额和金额拆分

### 7.1 可用账号

账号只有同时满足以下条件才参与当前批次：

- 账号状态为 enabled；
- 投注顺序有效；
- `perBetLimit > 0`；
- 币种为 CNY，金额精度为整数；
- 没有 active/unknown 账号锁；
- 登录成功，且执行前可以取得新鲜的网站余额；
- 网站返回的虚拟余额大于 0。

账号凭据继续使用现有本机加密存储，不进入日志、API 响应或审计明文。

### 7.2 可执行余额

每次准备使用账号时，通过该账号当前 session 刷新皇冠账户摘要。仅接受来源明确、币种为 CNY、可解析且非负的 `reportedBalance`。该值转换为整数 CNY 后作为本次分配的可执行余额。

缓存余额不能单独授权一次投注。余额刷新失败、币种不符、值缺失或时间不新鲜时跳过该账号，不调用 `FT_bet`。

### 7.3 拆分公式

单个账号本批次的计划金额：

```text
accountStake = min(
  batchRemaining,
  account.perBetLimit,
  freshReportedBalance,
  FT_order_view 返回的服务端最大可投金额
)
```

金额还必须不低于 `FT_order_view` 返回的最小金额，并满足服务端 stake step。无法满足最小金额或步长时跳过该账号。

示例：目标总金额 120 CNY，两个账号余额均为 1000 CNY、单笔上限均为 50 CNY。程序依次提交 50、50，剩余 20 因没有更多账号而记为未完成，不重复使用同一账号补投。

账号在一个批次内最多使用一次；明确成功或拒绝后解除锁，可以在后续独立规则批次中再次使用。

## 8. 皇冠 API Provider

### 8.1 会话和公共请求约束

Provider 必须复用现有投注账号登录管理器，并使用该账号独立 cookie/session。每次网络阶段前后都校验 executor lease 和 fencing token，防止旧 worker 在接管后继续提交。

只允许配置中的精确 HTTPS 皇冠 origin。禁止重定向到未允许 origin；禁止把 cookie、密码、完整请求体、完整响应体或注单敏感字段写入普通日志。

### 8.2 `FT_order_view`

预览请求必须由锁定的 canonical selection 映射生成，不能直接接收调用者任意字段。响应需要严格验证：

- 请求对应的赛事、市场、比赛阶段、方向和盘口仍一致；
- 返回赔率可解析，且仍在规则允许范围；
- 最小金额、最大金额和 stake step 可解析；
- 响应没有未知关键字段漂移或登录失效标记。

预览只读，不创建注单。任何复核失败都不得继续调用 `FT_bet`。

### 8.3 `FT_bet`

提交参数只能来自同一次成功 `FT_order_view` 的受信结果、锁定 selection、计划金额和当前 session。一次子注单只能发送一次 `FT_bet`。

结果分类：

| 分类 | 判断 | 后续 |
|---|---|---|
| accepted | 明确成功状态，并取得可持久化的皇冠注单标识 | 原子保存 accepted、已投金额和注单摘要，解除账号锁 |
| rejected | 明确业务拒绝，且确认没有创建注单 | 原子保存 rejected 和错误码，解除账号锁，将未投金额交给下个账号 |
| unknown | 超时、断线、响应截断、无法验证字段或无法确认是否创建注单 | 原子保存 unknown，保持账号锁，禁止自动重投 |

所有错误信息只保存稳定、脱敏的错误码。HTML、token、cookie、账号密码和原始响应不得进入可读审计。

### 8.4 Capability 调整

现有生产 `preview/submit/reconciliation = 0/0/0` 不能通过修改 UI 开关绕过。实现时应：

- 为当前皇冠协议支持的 exact market family 建立严格 request/response contract fixture；
- 只有 `FT_order_view` 和 `FT_bet` 字段映射、响应分类和金额语义通过 contract tests 的 market family 才允许执行；
- runtime 启动条件改为要求 preview 和 submit 可用，不再要求结算 reconciliation；
- 未证明或发生字段漂移的 market family 保持 fail-closed；
- unknown 通过持久锁防重复，不依赖自动 reconciliation 才能保证一次提交。

## 9. 账本、锁和恢复

继续复用现有 bet batch、child order、submit attempt、account lock、market-once 和 runtime lease 表。需要保证：

- 创建批次、认领市场和生成子注单是幂等的；
- 调用网络前已持久化唯一 submit attempt；
- accepted/rejected/unknown 与账号锁状态在同一事务中更新；
- worker 重启后 accepted/rejected 子注单不会重放；
- unknown 子注单保持终态未知和账号锁，不进入自动 retry queue；
- 每日完全重置可以按现有明确操作清除测试运行历史和未知锁；普通启动不能隐式清除。

拒绝后的换账号只处理当前仍未完成的目标金额，不把已经 accepted 或 unknown 的金额再次分配。

## 10. Watcher 稳定性修复

### 10.1 登录新鲜度

`monitor-login-not-fresh` 的根因是数据库只记录初次登录结果时间，而后续成功的 session 校验或 `get_game_list` 没有刷新该时间。

修复后，以下任一受信成功结果都刷新监控登录证明：

- 完整登录成功；
- 当前 session 的 `chk_login` 成功；
- 当前 session 的 `get_game_list` 成功并通过响应校验。

失败结果不能刷新成功时间。运行控制台以进程、lease、赔率新鲜度和登录证明四项共同判断 Watcher 是否就绪。

### 10.2 子进程退出与恢复

Monitor controller 记录最近退出时间、exit code、signal 和脱敏 stderr 摘要，并在非人工停止时执行有限自动恢复：

- 第 1 次：2 秒后重启；
- 第 2 次：5 秒后重启；
- 第 3 次：15 秒后重启；
- 连续 3 次仍失败：停止自动恢复，在运行控制台显示明确原因和人工“启动监控”入口。

人工停止、程序关闭、每日重置或更新安装期间不自动重启。启动前继续检查单实例 lease，不能产生两个 Watcher。

## 11. UI 与 API

不新增业务配置页面。现有页面调整如下：

- 投注规则：继续配置联赛、市场条件和目标总金额。
- 投注账号配置：继续配置账号、启用状态、顺序和单笔上限；账号检测显示最新虚拟余额及更新时间。
- 运行控制台：显示 Watcher 的进程/lease/登录/赔率状态，显示 worker 是否运行、最近批次和 accepted/rejected/unknown 数量。
- 投注历史：显示规则批次、各账号拆分金额、皇冠注单标识摘要和未完成金额。

不添加动态资格卡、独立预算、短期授权或结算操作。全局启动/停止仍保留，避免程序启动后未经用户操作立即下注。

API 只暴露脱敏业务状态。账号余额、注单标识和错误详情遵循现有本机访问和 CSRF 保护；账号密码、cookie 和皇冠请求材料永不返回前端。

## 12. 错误处理

| 场景 | 处理 |
|---|---|
| Watcher 未运行或数据过期 | 不创建新投注；运行控制台显示具体未就绪项 |
| 登录失效 | 尝试现有登录流程；失败则跳过账号，不提交 |
| 余额刷新失败 | 跳过账号，保留批次剩余金额给后续账号 |
| 当前盘口或赔率漂移 | 终止该批次未提交部分，不追逐新盘口 |
| `FT_order_view` 业务拒绝 | 跳过当前账号并尝试后续账号 |
| `FT_bet` 明确拒绝 | 解锁并换账号 |
| `FT_bet` 超时/断线/模糊响应 | 记 unknown、保持锁、停止该账号的任何自动提交 |
| worker 崩溃 | 依靠已持久化 attempt 和锁恢复，不重复发送已进入提交阶段的请求 |
| 字段契约漂移 | 对受影响 market family fail-closed，其他已验证 family 可继续 |

## 13. 验证与验收

### 13.1 自动化测试

实现必须先补失败测试，再写代码，覆盖：

- Watcher 登录证明刷新、异常退出记录、2/5/15 秒重启和人工停止不重启；
- 所有已支持 market family 的 selection 到 `FT_order_view`/`FT_bet` 字段映射；
- 余额 1000、单笔上限 50、不同目标总金额的拆分；
- 余额不足、服务端最小/最大金额、stake step；
- accepted 解锁并保存注单；
- rejected 解锁、换账号和无账号时放弃剩余金额；
- unknown 不重投且保持账号锁；
- 重复信号、重复消费、worker 重启和 lease 接管均不重复提交；
- 响应字段漂移、未允许 origin、重定向、登录失效全部在提交前阻断；
- 后端测试、前端测试、语法检查和生产构建。

### 13.2 本地虚拟账号验收

自动化测试通过后，使用用户已提供的皇冠虚拟余额账号做受控验收：

1. 确认每账号网站余额为 1000 CNY、单笔上限配置为 50 CNY。
2. 启动 Watcher 和自动投注 worker。
3. 仅等待真实监控变化触发现有规则，不人工伪造生产信号。
4. 连续验证 5 个 accepted 子注单；每笔必须有规则、盘口快照、账号、金额和皇冠注单标识对应关系。
5. 验证账号在 accepted 后解除锁，并能在后续独立批次复用。
6. 若出现 rejected，确认换账号；若出现 unknown，立即停止该账号后续提交并保留锁。
7. 完成 5 笔后停止自动投注并汇总触发、拆分、成功、拒绝、未知和未完成金额。

验收不等待比赛结束，不以网站结算结果作为本阶段完成条件。

## 14. 上线边界

- 首次启用前必须完成协议 contract tests 和一笔受控虚拟账号 smoke test。
- 只有明确 accepted 响应才能计入成功笔数。
- 任何 unknown 都优先防重复，不以提高完成率为由重试。
- 当前设计只针对不可充提的虚拟余额账号；如果以后接入可充值或可提现账号，必须重新设计风险、授权、预算和审计，不能直接沿用本设计自动启用。
