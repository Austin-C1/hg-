# 皇冠安全开工与运行控制台设计

## 目标

把每日开工流程收敛成可理解、可验证的四段链路：赔率监控、策略规则、投注账号、全局真实投注。每日完全重置后自动恢复赔率监控，但投注账号保持暂停、全局真实投注保持关闭，必须由用户检查后手动开启。

## 已确认决策

- 运行控制台采用“清爽操作台”方案，弱化装饰性卡片和大面积告警色。
- 每日完全重置完成后，只要主监控账号已启用且凭据存在，就自动启动 watcher。
- 安全开工模式：投注账号保持暂停，规则配置保留，全局真实投注保持关闭。
- “投注账号启用”只表示账号可以参与订单分配，不等于开启真实投注。
- 真实订单只有在规则允许、至少一个投注账号可分配、全局真实投注开启且所有安全预检通过时才允许提交。
- Crown canonical preview/submit/reconciliation 能力证据不足时继续 fail-closed，不通过 UI 文案或按钮绕过。

## 启用链路

```text
赔率监控在线
  -> 至少一条规则开启监控
  -> 规则允许真实投注
  -> 至少一个投注账号启用且检测通过
  -> 运维开启全局真实投注
  -> runtime 安全预检通过后运行
```

工作台必须逐层显示状态与阻断原因，避免用户把“账号启用”“规则允许”和“全局开关”理解为同一个动作。

## 每日完全重置

重置顺序：

1. 停止真实投注 worker。
2. 停止受管 watcher。
3. 清理点击前生成的赔率、监控状态、候选、投注账本、幂等锁、pending/unknown、对账、日志、索引与普通缓存。
4. 保留监控账号、投注账号凭据、登录会话、规则、Telegram、协议证据与运行依赖。
5. 将所有投注账号置为 `paused`，清除不应跨日继承的执行锁状态。
6. 强制将全局真实投注意图设为关闭。
7. 若监控账号已启用且凭据存在，启动 watcher。
8. 返回自动启动结果；工作台刷新后显示登录状态、watcher 心跳和赔率新鲜度。启动失败时展示明确原因。

## 工作台信息架构

### 顶部结论区

- 页面标题下直接显示“系统可监控”“等待安全开工”或“需要处理”。
- 主操作只随状态出现一个：监控停止时显示“启动监控”，运行时显示次要的“停止监控”。
- “立即刷新”保留为弱按钮。

### 下注准备链路

横向四步，在窄屏改为纵向：

1. 赔率监控：账号登录、watcher 唯一心跳、赔率新鲜度。
2. 策略规则：监控开启数、真实允许数。
3. 投注账号：可分配数、暂停数、检测失败数。
4. 全局真实投注：关闭、等待预检、运行或阻断。

每步只显示结论和一条可执行提示，并链接到对应配置页。

### 按钮规则

- 监控停止：显示蓝色“启动监控”。
- 监控运行：显示描边“停止监控”，停止前二次确认。
- 全局真实投注关闭：仅显示“开启真实投注”。
- 全局真实投注运行：仅显示“停止真实投注”。
- 开启条件不足时按钮禁用，旁边显示具体缺失项，不只依赖 tooltip。
- 无 `unknown`、对账或通知积压时，不渲染红色风险面板，只显示紧凑的“当前无待处理风险”。
- 每日完全重置位于页面底部“维护工具”，使用危险操作样式和二次确认。

## 投注账号页面

- 卡片顶部明确显示“已暂停 / 可分配 / 暂停处理中 / 检测中”。
- “启用账号”旁显示说明：启用后只加入订单分配，不会自动开启全局真实投注。
- 已启用账号只显示“暂停账号”，不同时显示相反操作。
- 检测账号是独立动作；检测成功不自动启用账号。
- 删除保持危险操作，编辑和历史明细降为次要操作。

## 视觉原则

- 浅色工作区、单一蓝色主操作色。
- 绿色仅表示已验证正常，黄色表示需要操作，红色只用于真实风险、失败和不可撤销操作。
- 不使用大面积深色状态条，不为 0 风险使用红色背景。
- 减少卡片嵌套；用分组、细分隔线和状态文本建立层级。
- 桌面优先展示完整链路，390px 宽度下按钮占满可用宽度且不横向溢出。

## 验收标准

- 重置前 watcher 无论是否运行，重置后只要监控账号可用就会自动启动。
- 重置后两个投注账号均为暂停，全局真实投注关闭。
- 工作台能直接启动/停止监控，并显示 watcher 真实进程/心跳状态。
- 真实投注关闭时不显示“停止真实投注”；运行时不显示“开启真实投注”。
- 开启真实投注被阻断时，页面展示完整缺失条件。
- 0 风险时没有红色主面板。
- 投注账号页面能区分“账号可分配”和“全局真实投注”。
- 后端、前端测试、production build、桌面与 390px 浏览器验收全部通过。
